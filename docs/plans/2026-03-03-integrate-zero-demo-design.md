# IntegrateZero Demo Design

## Goal

End-to-end demo of the IntegrateZero system: supervised pretraining -> MCTS search -> RL self-play. Multi-step expression transformation with per-step derivative verification.

Medium simplification: ~12M parameter model, complete expression types, full MCTS. Skip curriculum learning and complex hyperparameter tuning.

---

## 1. Expression Representation

Prefix notation (Polish notation) for all mathematical expressions.

```
sin(x^2 + 1)         -> [sin, add, pow, x, 2, 1]
x*sin(x) + cos(x)    -> [add, mul, x, sin, x, cos, x]
INT sin(x) dx         -> [INT, sin, x, x]
                              ^^^^^^^^  ^
                              integrand  variable
```

INT has arity=2: (integrand, integration variable). Integration variable is always x.

### Vocabulary (~85 tokens)

| Category | Tokens |
|----------|--------|
| Operators | `add`, `sub`, `mul`, `div`, `pow`, `neg` |
| Functions | `sin`, `cos`, `tan`, `exp`, `log`, `sqrt`, `arcsin`, `arccos`, `arctan`, `sinh`, `cosh`, `tanh` |
| Integration variable | `x` |
| Real parameters | `a`, `b`, `c`, `d` (SymPy: `Symbol(real=True)`) |
| Integer parameters | `k`, `l`, `m`, `n` (SymPy: `Symbol(integer=True)`) |
| Numeric constants | integers `-10` to `10`, `pi`, `e` |
| Special | `INT` (arity=2), `PAD`, `BOS`, `EOS`, `SEP` |

Note: `l` (lowercase L) vs `1` (digit one) — use distinct internal string representations (e.g. `param_l` vs `const_1`) to avoid confusion.

### SymPy Interop

Bidirectional conversion module: `prefix_tokens_to_sympy()` and `sympy_to_prefix_tokens()`.

### General Position Assumption

For symbolic parameters, assume general position (no degenerate values). This is safe because:
- Verification only differentiates (forward direction), never integrates
- Training data is generated by "random F, then differentiate", naturally avoiding degenerate cases
- If the integrand is `1/x` (i.e. n=-1 case), the model learns to generate `log(x)` as a separate training example

## 2. Multi-Step Transformation

The model iteratively transforms expressions. Two types of valid steps:

| Step type | Effect | Verification |
|-----------|--------|-------------|
| Integration step | Reduces/eliminates INT symbols | `d/dx(B) == A` |
| Identity rewrite | Algebraically equivalent, keeps INT | `simplify(A - B) == 0` |

### Example

```
Step 0: INT x*cos(x) dx
Step 1: add, mul, x, sin, x, INT, sin, x, x     (= x*sin(x) - INT sin(x) dx)
        -> integration step, verify d/dx[x*sin(x)] - sin(x) = x*cos(x) OK
        -> still has INT, continue
Step 2: add, mul, x, sin, x, cos, x              (= x*sin(x) + cos(x))
        -> integration step, verify d/dx[x*sin(x) + cos(x)] = x*cos(x) OK
        -> no INT, done! reward = +1
```

### Verification Logic

```
Given state A and generated B:
1. Check simplify(A - B) == 0 -> identity rewrite, legal, reward = 0
2. Check derivative verification -> integration step, legal, reward = 0 (mid) or +1 (terminal)
3. Neither -> illegal, reject

Termination: B has no INT and verification passes -> reward = +1
Timeout: exceed max steps (20) -> episode failure
```

### Reward Design

- Successful termination: +1
- Identity rewrite: 0
- Integration step (non-terminal): 0
- Illegal generation: rejected (not added to MCTS tree)
- Timeout: 0

## 3. Model Architecture

Decoder-only Transformer (GPT-style).

```
Input:  [BOS] A_tokens [SEP] B_tokens [EOS]
        ~~~~~~~~~~~~~ ~~~~~~~~~~~~~~~~
        current expr A  generated expr B

Transformer Decoder (8 layers, d_model=384, 6 heads, d_ff=1536)
  |
  |-- Generation: autoregressive prediction of tokens after [SEP]
  |                causal mask + teacher forcing during training
  |
  |-- Value Head: hidden state at [SEP] position -> MLP (384->256->1, sigmoid)
                  output v(A) in [0,1]: probability of solving from state A
```

### Parameters (~12M)

| Component | Params |
|-----------|--------|
| Decoder (8 layers, d=384) | ~12M |
| Value Head | ~0.1M |
| Embedding + Output Projection | ~0.1M |
| **Total** | **~12M** |

### Decoding Constraints

Maintain an arity stack during generation to guarantee syntactically valid prefix expressions:
- Generate `add` -> push: need 2 children
- Generate `sin` -> push: need 1 child
- Generate `x` or constant -> consume 1 slot
- Stack empty -> force `EOS`
- Stack not empty -> mask `EOS`

## 4. Training Pipeline

### Configuration

```
pretrain: bool = True   # True = supervised pretrain + RL (AlphaGo route, default)
                        # False = pure RL from random init (AlphaZero route)
```

This serves as an ablation experiment: compare convergence and final solve rate between the two routes.

### Phase 1: Data Generation + Supervised Pretraining (skipped if pretrain=False)

**Data generation**: randomly generate F(x), differentiate to get f(x), form (f, F) training pairs.

```
Random expression generator (controllable depth 2-8)
  |
  +-- Generate F(x) = x*sin(x) + cos(x)
  +-- Differentiate f(x) = x*cos(x)           (SymPy, trivial)
  +-- Training pair: (f, F)

Multi-step data:
  F2 = x*sin(x) + cos(x)             (final answer)
  F1 = x*sin(x) - INT sin(x) dx      (intermediate, manually insert INT)
  F0 = INT x*cos(x) dx               (original problem)
  Training pairs: F0->F1, F1->F2
```

Initial dataset: ~500K pairs, stratified sampling by expression depth.

**Supervised training**:
- Policy: standard cross-entropy loss with teacher forcing
- Value: binary classification loss (solvable=1 / unsolvable=0)

### Phase 2: MCTS + Self-Play RL

```
For each problem f(x):

1. MCTS Search:
   At state A, sample k=8 candidates B from Policy
     |-- Verify each B (identity / integration step / illegal)
     |-- Discard illegal ones
     |-- Evaluate legal ones with Value v(B)
     |-- UCB selection:
         score = v(B) + c_puct * P(B) * sqrt(N_parent) / (1 + N_child)
   Repeat until solution found, max steps (20), or budget exhausted (200 nodes)

2. Collect training data:
   Success: (state, MCTS-selected action, reward=+1) for each step
   Failure: (state, *, reward=0)

3. Update network:
   Policy -> learn from MCTS improved probability distribution
   Value  -> learn from actual outcome (success=1, failure=0)
```

**Key hyperparameters**:

| Parameter | Value |
|-----------|-------|
| MCTS candidates per step k | 8 |
| Max solving steps | 20 |
| MCTS search budget | 200 nodes/problem |
| c_puct (exploration coefficient) | 1.5 |
| Training batch size | 256 |
| Learning rate | 1e-4 (Adam) |
| RL iteration rounds | 50-100 |
| Problems per round | 2000 |

### Phase 3: Not implemented in demo

Curriculum learning and automatic difficulty adjustment deferred. Train with fixed difficulty distribution.

## 5. Evaluation

### Test Sets

| Test set | Source | Count |
|----------|--------|-------|
| In-distribution | Same generation method, no overlap | 5,000 |
| Out-of-distribution | Deeper/more complex expressions | 2,000 |
| Textbook basics | Hand-collected: polynomial, trig, exp, integration by parts | ~100 |

RUBI 72,944 problems and Lample & Charton test sets as future targets.

### Metrics

| Metric | Description |
|--------|-------------|
| Solve rate | Fraction successfully integrated within max steps |
| Average steps | Number of transformation steps for successful solves |
| Verification pass rate | Fraction of generated expressions passing verification (generation quality) |
| MCTS vs Greedy | Solve rate comparison with/without search |

### Baselines

| Baseline | Description |
|----------|-------------|
| SymPy integrate() | Direct call to SymPy integrator |
| Supervised only | Phase 1 model, no RL |
| Supervised + Greedy | No MCTS, argmax at each step |
| Supervised + MCTS + RL | Full system |

## 6. Code Organization

Modular Python package with `pyproject.toml`:

```
integrate_zero/
  __init__.py
  data/           # expression generation, tokenizer, dataset
  model/          # transformer, value head
  mcts/           # MCTS search
  train/          # supervised and RL training loops
  eval/           # evaluation and benchmarking
  verify/         # SymPy derivative verification
```

Device-agnostic: support CUDA, MPS (MacBook), and CPU.
